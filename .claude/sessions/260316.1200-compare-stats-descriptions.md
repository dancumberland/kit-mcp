## Context
Dan tested the Kit MCP by asking another Claude session (claude.ai Max plan) to "find my most engaged 100 subscribers who have been on my list for more than 6 months." The Claude session failed — it tried `list` with sort_field, checked segments, fell back to tags as a proxy, and never discovered the `stats` endpoint could provide actual engagement numbers. The result was a list sorted by nothing meaningful.

## Objectives
1. Add `compare_stats` action for batch engagement stat fetching and ranking
2. Rewrite tool descriptions to guide Claude toward the correct workflow
3. Prevent the failure mode where Claude uses `list`/`filter` for engagement data

## Implementation Details

### compare_stats action (subscribers.ts)
- Schema: `ids: z.array(z.coerce.number()).min(2).max(100)` — accepts up to 100 subscriber IDs
- Handler processes IDs in **batches of 5** via `Promise.allSettled` to avoid rate limiter race conditions (SlidingWindowRateLimiter.acquire() has no mutex)
- Each subscriber requires 2 API calls: `GET /subscribers/{id}` + `GET /subscribers/{id}/stats`
- Gracefully handles per-subscriber failures (404, timeout) without aborting the batch
- Tracks total API calls for the response footer

### formatSubscriberComparison (formatters.ts)
- Sorts results by open_rate descending (most engaged first)
- Ranked output: `1. Name <email> — Open: X% | Click: Y% | Sent: N | Last open: date (ID: N)`
- Shows skipped IDs with error reasons
- Footer with API call count
- Handles all-failed case with clear error message

### Description rewrite (server.ts)
Two iterations:
- v1.3.0: Added `compare_stats` to action list and noted `list`/`filter` don't return engagement data
- v1.3.1: Rewrote to lead with `WORKFLOW FOR ENGAGEMENT RANKING:` prescribing the exact 2-step process, plus explicit "Never use 'list' or 'filter' to find engaged subscribers"

### Schema description (subscribers.ts)
- v1.3.0: Had anti-loop warning "Do NOT loop this action over hundreds of subscribers" — this directly conflicted with the use case
- v1.3.1: Replaced with positive guidance: "Step 1: use engagement_filter to find candidate IDs. Step 2: pass those IDs here to get ranked engagement stats."

## Key Decisions
1. **Max 100, not 25**: Original plan capped at 25 to avoid rate limit issues. Real-world testing showed 25 is too small for "find my top 100" requests. 100 subscribers × 2 calls = 200 requests, processed in 20 batches of 5. Feasible within 120 req/min limit.
2. **Removed anti-loop warning**: The red team's recommendation to warn against looping was counterproductive — it discouraged the exact workflow the feature was built for.
3. **Prescriptive over descriptive**: The v1.3.0 description listed capabilities but didn't tell Claude *when* to use them. v1.3.1 leads with the workflow recipe because that's what Claude needs to avoid the proxy-fallback failure mode.
